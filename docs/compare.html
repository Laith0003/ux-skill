Scoring methodology

Catalog depth = entries shipped in structured data. Linter = deterministic rule enforcement (regex, AST). Discovery = forcing-function intake before generation. Pipeline = number of chained commands. Brand specs = named brand DESIGN.md spec files. Motion = number of named motion presets. Cross-IDE = number of IDEs supported. A11y = explicit accessibility command + WCAG references. Activity = days since last push. Docs = README depth + examples + screenshots.

1 = absent / 2 = mentioned / 3 = present / 4 = mature / 5 = best-in-class. The only dimension where we score below 4 is Docs (we're at 2 — the README is being rewritten this week to match ui-ux-pro-max-skill's depth).

Common questions about this comparison

How were the scores in each cell determined?
Each cell is scored 1 (absent or broken) to 5 (best in class) against a concrete rubric for that dimension. For catalog depth the score reflects the number of queryable brand and pattern entries. For the linter it reflects whether rules fire deterministically on real output, how many rules exist, and whether severity levels are assigned. Stars are pulled live from gh api and frozen at the date shown. No cell is left blank and no dimension was chosen to flatter ux-skill.
Why does ux-skill score lower on GitHub stars than the top competitors?
ui-ux-pro-max-skill launched roughly nine months before ux-skill reached v3.0.0-stable. Stars accumulate with time and distribution; the scorecard measures what is inside the repo, not how long it has been visible. The star gap is real and acknowledged in the disclosure box at the top of this page.
Does ux-skill work in IDEs other than Claude Code?
Yes. The plugin is listed as compatible with 17 AI coding environments including Cursor, Windsurf, Zed, JetBrains AI, GitHub Copilot, and VS Code with Continue. The MCP server path works in any host that speaks the Model Context Protocol. The CLI (pip install uxskill) is IDE-agnostic.
What makes the ux-skill linter different from running a generic CSS linter?
Generic CSS linters catch invalid syntax and property typos. The ux-skill linter catches AI-slop fingerprints: valid CSS that still reads as generated because it matches the centroid of training data. Examples include a violet-to-indigo gradient with no brand rationale, Inter as the only typeface when a display family would add character, or three feature cards at identical visual weight. These are correct code that is wrong design. No generic linter catches them.
Is the scorecard updated as competing plugins ship new features?
Star counts are refreshed each release cycle from gh api; the date frozen is shown in the footer. Feature scores are reviewed when a competing plugin ships a significant release. If a rival ships a deterministic linter or a 100+ brand spec catalogue, those cells will move. The methodology is documented at the bottom of this page so you can verify any cell independently.
