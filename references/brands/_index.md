# Brand library — 160 production-grade DESIGN.md files

When a user asks for a specific brand's aesthetic — "build me a landing in Stripe's style", "match the Notion editorial feel", "make it like Linear" — read the matching file in this directory. Every brand here has a full design system spec: colors, typography, layout, components, motion, content tone, do/don't pairs.

The frontend-engineer sub-agent MUST consult the relevant brand file before producing output when the user has named a specific brand. The brand's design language overrides defaults; the plugin's anti-AI-slop discipline still applies on top.

## How to use

1. **User names a brand**: the user's brief includes "make it look like X" or "match the Y aesthetic" or pastes a brand reference.
2. **Match against this index**: find the closest brand in the list above. Substring match is fine ("stripe" → `stripe.md`).
3. **Read the full brand file**: pull the complete DESIGN.md spec for that brand.
4. **Pass to frontend-engineer**: include the brand spec verbatim in the sub-agent's prompt. The brand becomes the visual ground truth; the plugin's anti-slop discipline applies on top (no AI fingerprints, full SEO surface, etc.).
5. **Echo back**: confirm to the user which brand the output is matching.

## When the user names a brand NOT in the library

Offer to:
- **Adapt the closest match** — e.g., user wants "Mailchimp aesthetic" → propose adapting Intercom (closest in spirit) or Notion (closest in warmth).
- **Run discovery on the brand** — ask the user for 3-5 references of that brand's surfaces and build a custom brand spec inline.
- **Use the closest category default** — pick from a generic style library (`references/styles/library.md`).

## Updating the library

To refresh a brand's spec to the latest version:
```
npx getdesign@latest add <brand>
```
Then commit the updated `references/brands/<brand>.md`.

To add a new brand not in this catalog, propose a PR with a complete DESIGN.md spec following the format used in the existing brand files.
